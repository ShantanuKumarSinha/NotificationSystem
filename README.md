# NotificationSystem

`NotificationSystem` is a demo Spring Boot service that publishes notification events to Kafka, consumes them, and dispatches them to a channel-specific handler.

It is currently best understood as a **Kafka integration sample** rather than a production-ready notification platform.

## What this project demonstrates

- Producing `NotificationEvent` messages to Kafka
- Consuming events with Spring Kafka listeners
- Basic retry and dead-letter-topic (DLT) handling
- Simple duplicate suppression using Caffeine cache
- Channel-based dispatching for `EMAIL`, `SMS`, and `PUSH_NOTIFICATION`
- Actuator and Prometheus metrics exposure

## Tech stack

- Java 21
- Spring Boot 4.0.3
- Spring Web MVC
- Spring for Apache Kafka
- Spring Boot Actuator
- Micrometer Prometheus registry
- Caffeine cache
- Gradle
- Docker Compose for local Kafka

## Project flow

1. `POST /notifications` is called.
2. `NotificationController` creates a **demo** `NotificationEvent` in code.
3. `NotificationProducer` publishes the event to Kafka topic `notification.events`.
4. `NotificationConsumer` reads the event from Kafka.
5. The consumer skips recently seen `eventId` values using an in-memory Caffeine cache.
6. `NotificationDispatcher` routes the event to a channel-specific method.
7. If listener processing keeps failing, Spring Kafka sends the record to `notification.events.DLT`.

## Current API behavior

### `POST /notifications`

This endpoint currently **does not accept a request body**. It always creates a hardcoded demo notification event and returns the generated `eventId`.

Response:

- Status: `202 Accepted`
- Body: plain text event ID

Example:

```bash
curl -i -X POST http://localhost:8080/notifications
```

Typical response:

```http
HTTP/1.1 202 Accepted
Content-Type: text/plain;charset=UTF-8

0c44c641-0a1d-4f1e-b1f6-6d1176f4f8a2
```

The event currently generated by the controller has this shape:

```json
{
  "eventId": "<generated UUID>",
  "userId": "user123",
  "message": "This is a test notification.",
  "channel": "EMAIL",
  "title": "Test Notification",
  "body": "This is a test notification.",
  "metadata": {
	"key1": "value1",
	"key2": "value2"
  },
  "createdAt": "<current instant>"
}
```

## Domain model

### `NotificationEvent`

Kafka messages use this record:

- `eventId`: unique event identifier
- `userId`: target user
- `message`: message summary
- `channel`: one of `EMAIL`, `SMS`, `PUSH_NOTIFICATION`
- `title`: notification title
- `body`: notification body
- `metadata`: arbitrary key/value pairs
- `createdAt`: event creation timestamp

### `NotificationRequest`

The project also contains a `NotificationRequest` record, but it is **not used by the controller yet**.

## Kafka topics and listeners

Topics declared in `TopicConfig`:

- `notification.events` with 6 partitions
- `notification.events.DLT` with 2 partitions

Listener bindings:

- Main consumer listens to `${topics.notifiications}`
- DLT consumer listens to `${topics.notifications-dlt}`

Configured values in `application.yml`:

- `topics.notifiications: notification.events`
- `topics.notifications-dlt: notification.events.DLT`

> Note: the property key is spelled `notifiications` in the code and config. The spelling is unusual, but it is currently consistent and therefore functional.

The producer currently publishes to a hardcoded topic name:

- `notification.events`

## Retry and dead-letter behavior

`KafkaConfig` registers a `DefaultErrorHandler` with:

- `5` max retries
- initial backoff: `1000 ms`
- multiplier: `2.0`
- max backoff: `10000 ms`

Records that still fail are published to:

- `<original-topic>.DLT`

`IllegalArgumentException` is marked as non-retryable.

## Duplicate suppression

`NotificationConsumer` keeps an in-memory Caffeine cache of processed event IDs:

- expiry: 10 minutes
- maximum size: 10,000 entries

If an `eventId` is already present in the cache, the event is skipped.

## Dispatch behavior

`NotificationDispatcher` currently routes events by channel and writes a message to standard output:

- `EMAIL` -> `sendEmail(...)`
- `SMS` -> `sendSms(...)`
- `PUSH_NOTIFICATION` -> `sendPushNotification(...)`

There is no real email, SMS, or push provider integration yet.

## Configuration

Key runtime settings from `src/main/resources/application.yml`:

- Kafka bootstrap server: `localhost:9092`
- Consumer group ID: `notification-service`
- Consumer offset reset: `earliest`
- Listener concurrency: `3`
- Producer acks: `all`
- Producer idempotence: enabled
- Actuator exposure: `health`, `info`, `metrics`, `prometheus`

## Limitations and caveats

- `POST /notifications` is demo-only and does not consume `NotificationRequest`.
- The producer topic is hardcoded in `NotificationProducer` instead of being injected from configuration.
- The property name `topics.notifiications` contains a typo, so any cleanup must be coordinated across config and listeners.
- The DLT topic is configured with **2 partitions** while the main topic has **6 partitions**. Because the dead-letter recoverer republishes to the same partition number on `<topic>.DLT`, this partition mismatch is a risk for records coming from partitions that do not exist on the DLT.
- `NotificationDispatcher` only logs to the console; no external notification providers are integrated.
- Logging is done with `System.out.println` / `System.err.println` rather than a structured logging framework.

## Suggested next improvements

1. Accept `NotificationRequest` as a request body in the controller.
2. Build `NotificationEvent` from client input instead of generating a demo payload.
3. Externalize the producer topic name into configuration.
4. Align DLT partition count with the main topic.
5. Replace console logging with structured application logs.
6. Add integration tests for Kafka publishing, retries, and DLT routing.
